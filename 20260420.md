# TokenSkip pipeline — findings & empirical notes

*Scope: work on `/workspace/LatentReasoning` between the first run of `scripts/sft/pipeline_tokenskip_math.sh` and the eval that followed `max_steps=1200` training. Covers pipeline modifications, training results, disk/runtime costs, and a small empirical study of how LLMLingua-2's compression decisions relate to language-model entropy.*

---

## 1. Pipeline modifications

Starting from the paper-aligned N=1 / greedy setup in `pipeline_tokenskip_math.sh`, the following changes were made:

| Change                      | File(s)                                                      | Rationale                                                    |
| --------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **Multi-sample collection** | `sft/methods/tokenskip/collect.py`, pipeline script          | Added `--num-samples` / `--temperature`; default `NUM_SAMPLES=8 @ T=0.7`. Each sample gets a unique `source_id` (`_s{idx}`) and carries `question_source_id` + `sample_index` metadata. → ~8× more training data. |
| **Fan-out in `select.py`**  | `sft/methods/tokenskip/select.py`                            | Iterate over `original_records.values()` instead of `raw_data`, so N candidate CoTs per question naturally produce N training rows. Random ratio still sampled per row. |
| **Sharded compression**     | `sft/methods/tokenskip/compress.py`, pipeline script         | Added `--num-shards` / `--shard-id`; new `COMPRESS_SHARDS` env var. Step 2 parallelises across 4 GPUs → ~4× wall-clock speedup. |
| **Added `ratio=0.1`**       | pipeline script                                              | Now: `COMPRESS_RATIOS=0.1,0.3,0.5,0.7`, `TRAIN_RATIOS=1.0,0.1,0.3,0.5,0.7`, `EVAL_RATIOS=1.0,0.7,0.5,0.3,0.1`. |
| **Training config tweaks**  | `config/sft/math_tokenskip_rawtext.yaml`, `sft/train.py`     | `lr_scheduler_type: cosine → constant_with_warmup` (no decay after warmup), `save_steps: 100 → 300` (couples to `eval_steps` via `load_best_model_at_end`), new `max_steps: 1200`. |
| **Gradio compression demo** | `tools/compression_demo/app.py`, `scripts/demo/run_compression_ui.sh` | Interactive inspector for LLMLingua-2 with Samples / Custom / LM-Entropy tabs. |

## 2. Dataset sizing (after the changes)

Funnel, with `NUM_SAMPLES=8`, `MAX_COT_TOKENS=2000`, Qwen3-4B-Thinking-2507 on MATH train (7,500 questions):

| Stage                | Filter                                                       | Approx count                |
| -------------------- | ------------------------------------------------------------ | --------------------------- |
| Collect              | —                                                            | 7,500 × 8 = **60,000** rows |
| Compress             | `is_correct ∧ len ≤ 2000`                                    | **~41,000** per ratio       |
| Prepare              | one ratio sampled per row (5 ratios in the pool); ratio=1.0 survives correctness, ratio<1.0 survives length | **~44,000** SFT rows        |
| Train 90% / Eval 10% | HF `eval_ratio`                                              | ~40k train / ~4k eval       |

**One SFT sample per collected CoT** — not one per ratio. Each row carries its own randomly-sampled ratio; this matches the TokenSkip paper's training scheme.

## 3. Compression rate is a soft target

Tested LLMLingua-2's actual vs. target rate by running `PromptCompressor.compress_prompt(text, rate=r)` on four math-style CoTs at r ∈ {0.3, 0.5, 0.7}:

| target   | sample lengths → actual rate                             |
| -------- | -------------------------------------------------------- |
| **0.50** | 28→**0.500**, 52→**0.615**, 128→**0.500**, 106→**0.594** |
| 0.30     | 28→0.286, 52→0.365, 128→0.336, 106→0.396                 |
| 0.70     | 28→0.714, 52→0.750, 128→0.703, 106→0.755                 |

**Mean |actual − target| ≈ 0.05, worst case ≈ 0.115.**

The rate is not a hard top-k cut. Primary causes (documented in the demo's in-app accordion):

1. **Threshold, not top-k.** LLMLingua-2 picks a probability threshold on per-token keep-scores that aims for the requested rate; if scores cluster near the threshold, the achieved count drifts.
2. **Chunking.** Long prompts are split into ~512-token windows and the rate is applied per chunk.
3. **Tokenizer mismatch.** Rate is defined over xlm-roberta tokens, not the downstream LM's tokens — re-tokenization shifts the ratio again.
4. **Asymmetric bias.** Empirically the model overshoots (keeps slightly more than requested) more often than it undershoots.

The training pipeline already stores `actual_compression_rate` alongside the target (`compress.py:122`), so SFT labels use the *achieved* rate — not the requested one.

## 4. Training results on MATH-500

`max_steps=1200`, `constant_with_warmup` LR at 5e-5, 4× GPU training, ~44k SFT rows (so 1200 steps ≈ **0.87 epochs**).

### Accuracy and reasoning-length by eval ratio

| Ratio   | Accuracy  | Avg think tokens | Avg tokens (correct only) |
| ------- | --------- | ---------------- | ------------------------- |
| 1.0     | 55.4%     | 2,322            | 1,569                     |
| **0.7** | **63.4%** | 1,693            | 1,175                     |
| 0.5     | 45.0%     | 2,088            | 721                       |
| 0.3     | 24.6%     | 2,947            | 786                       |
| 0.1     | 9.0%      | 3,759            | 1,689                     |

### Comparison vs. the previous N=1 greedy run (from README)

| Ratio | N=1 accuracy (old) | N=8 accuracy (new) | Δ         |
| ----- | ------------------ | ------------------ | --------- |
| 1.0   | 42.8%              | 55.4%              | **+12.6** |
| 0.7   | 52.6%              | 63.4%              | **+10.8** |
| 0.5   | 51.6%              | 45.0%              | −6.6      |
| 0.3   | 44.0%              | 24.6%              | −19.4     |
| 0.1   | —                  | 9.0%               | n/a       |

### What worked

- **`ratio=0.7` beats `1.0` by +8 points** — the paper's "compressed reasoning is more efficient" claim reproduces cleanly here.
- **High-ratio training benefits from the 8× data** — +12.6 points at r=1.0 over the N=1 baseline.

### What didn't — and why

On the face of it, low ratios (0.3 and 0.1) regressed hard compared to the old N=1 run, and average think-token count *increased* as compression got more aggressive (2,088 → 2,947 → 3,759). That looked like the compressed reasoning itself getting worse. A closer look at the eval dumps told a different story.

#### Failure mode: the model doesn't know when to stop

We inspected how often each generation hit the 4,096-token generation cap and how often it failed to emit the `</think>` closing tag at all:

| Ratio   | % hitting 4,096-token cap | % that never closed `</think>` |
| ------- | ------------------------- | ------------------------------ |
| 1.0     | 31.2%                     | 31.2%                          |
| **0.7** | **15.6%**                 | 15.4%                          |
| 0.5     | 40.4%                     | 40.4%                          |
| 0.3     | 68.8%                     | 68.8%                          |
| 0.1     | **91.6%**                 | **91.6%**                      |

At ratio=0.1 **more than 9 in 10** generations are run-on: the model keeps producing compressed-style text until the token cap and never transitions to the answer. The 0.7 "sweet spot" is as much about having the lowest run-on rate as it is about the compression quality itself.

#### Conditioning on clean termination reshapes the picture

Splitting overall accuracy by whether the model terminated the thinking block cleanly:

| Ratio | N    | Terminated | P(correct \| terminated) | P(correct \| run-on) |
| ----- | ---- | ---------- | ------------------------ | -------------------- |
| 1.0   | 500  | 344        | **77.9%**                | 5.8%                 |
| 0.7   | 500  | 423        | **74.5%**                | 2.6%                 |
| 0.5   | 500  | 298        | **74.5%**                | 1.5%                 |
| 0.3   | 500  | 156        | **69.9%**                | 4.1%                 |
| 0.1   | 500  | 42         | **64.3%**                | 3.9%                 |

**When the model manages to close `</think>`, it's 64–78% accurate across *all* ratios** — even at ratio=0.1, a compression the model has barely been trained on, terminated reasoning is 64% correct (comparable to the 0.7 "sweet spot"). Run-on outputs are essentially random.

So the low-ratio regression is **a termination failure, not a reasoning failure**. The model has learned how to produce compressed reasoning; it hasn't learned how to end it.

#### Examples

**Good case — `math_test_5` (hexagon perimeter question).** Correct at every ratio, with think-token counts decreasing monotonically `671 → 594 → 202 → 193 → 55`:

> *(ratio=0.1, 55 think tokens, correct)*
> `hexagon six triangles perimeter 21 inches hexagon six triangles sides hexagon equal triangles perimeter 21 inches 21 divided 3 7 inches hexagon six triangles 6 42 inches perimeter 21 inches 7 inches 42 inches`

Telegraphic but coherent: concept → numbers → computation → answer. The model clearly *can* compress at ratio=0.1.

**Regression case — `math_test_3` (divisors of 196).** Correct at ratio=0.7, fails at 0.5 / 0.3 / 0.1.

*At ratio=0.7 (586 tokens, correct):*

> ```
> problem is asking how many positive whole-number divisors 196 has. need to remember find number of divisors factorize number into prime factors ... 14 is 2*7. 14 squared is (2* ...
> ```

*At ratio=0.3 (4,096 tokens cap, run-on, wrong — prediction has no `</think>` tag):*

> `divisors 196 factorize 196 prime factors ... 196 = 2^2 * 7^2. divisors (2 + 1) * (2 + 1) = 3 * 3 = 9. 1 2 4 7 14 28 49 98 196 10. 196 14 squared 14 2*7 14 squared 2^2 * 7^2. divisors (2 + 1) * (2 + 1) = 9. 196 14 squared 14 2*7 14 squa` *[truncated at max_tokens]*

The reasoning *gets the answer* (`= 9`, and lists the nine divisors correctly), but then falls into a loop instead of writing the closing `</think>` and producing a final `\boxed{9}`. The answer extractor then picks up a stray `10` from the repetition and scores the sample wrong.

*At ratio=0.1 (109 tokens, wrong):*

> ```
> divisors 196 divisors 196 196 14 14 196 196 2 2 7 7. 2^2 7^2 divisors 196 196 1 2 4 7 14 28 49 98 196 10. 196 10. 196 10. 196 10.
> ```

Same behavior in extreme form — the model drops into a repetition loop (`196 10.` four times) before producing a proper answer.

#### Why this happens

`max_steps=1200` × 32 samples/step = 38.4k samples seen — roughly **0.87 epochs** through the 44k-row SFT set. The model saw each training example less than once on average. The compressed-style tokens appear frequently enough to imprint the *style*, but the `</think>` terminator token appears only once per training example; under-exposure to that single-position signal at aggressive ratios seems to be exactly the bottleneck.

Upshot: the fix is almost certainly **more training steps** — not different data, different scheduler, or a different LR. A follow-up run with `max_steps≈4000` (≈ 3 full epochs) should let the model learn when to emit `</think>` at low ratios, at which point the ~65–75% "conditional" accuracy should flow through to the "overall" column.

### Batch=128 follow-up (~3.5 epochs) — directly testing the "more training" hypothesis

We reran the same pipeline with `gradient_accumulation_steps: 8 → 32` (1 × 32 × 4 GPUs = **128 global batch**) and kept `max_steps=1200`. That raises total samples seen from 38,400 (0.87 epochs) to **153,600 (≈ 3.5 epochs)** — the cleanest test of the "more training" hypothesis.

#### Overall accuracy

| Ratio | batch=32, 0.87 epochs | batch=128, ~3.5 epochs | Δ |
|---|---|---|---|
| 1.0 | 55.4% | **55.8%** | +0.4 |
| **0.7** | 63.4% | **65.8%** | **+2.4** |
| 0.5 | 45.0% | **48.2%** | +3.2 |
| 0.3 | 24.6% | **27.4%** | +2.8 |
| 0.1 | 9.0% | **9.2%** | +0.2 |

#### Termination breakdown

| Ratio | Terminated (old → new) | P(correct \| terminated) | P(correct \| run-on) |
|---|---|---|---|
| 1.0 | 344 → 347 (69% → 69%) | 77.9% → 77.2% | 5.8% → 7.2% |
| 0.7 | 423 → 431 (85% → 86%) | 74.5% → 75.6% | 2.6% → 4.3% |
| 0.5 | 298 → 307 (60% → 61%) | 74.5% → 76.2% | 1.5% → 3.6% |
| 0.3 | 156 → 160 (31% → 32%) | 69.9% → 74.4% | 4.1% → 5.3% |
| 0.1 | 42 → 54 (8% → 11%) | 64.3% → **50.0%** | 3.9% → 4.3% |

#### What changed

- **ratio=0.7 improved by +2.4 points** to a new best of **65.8%**. Middle ratios (0.5, 0.3) also gained ~2–3 points.
- **Termination rates at ratio=0.1 barely moved** (8% → 11%). The 4× training budget produced only a ~3-point improvement in termination where it matters most.
- **Conditional accuracy at ratio=0.1 actually got *worse*** (64.3% → **50.0%**). With 4× more exposure the model learned a different, worse way to terminate at that ratio — possibly over-fitting on the very short (~200-token) ratio=0.1 training distribution and truncating reasoning prematurely.

#### Takeaway — more training is not the fix

The "more epochs" hypothesis is falsified at ratio=0.1 and only weakly supported at middle ratios. ratio=0.7 gains +2.4 points, but the dominant failure mode (run-on → no `</think>`) at aggressive ratios is essentially untouched by scaling the batch×epochs budget 4×. The conditional-accuracy regression at ratio=0.1 is particularly telling — extra data on a thin, noisy distribution can hurt, not help.

This strengthens the case for the §7-item-2 intervention: **explicit upweighting of the `</think>` token position in the SFT loss**. The terminator signal is present in every training row but appears only once per row; at the most aggressive compression ratios the distribution of "when to stop" is too thin for standard next-token CE to learn it even at 3.5 epochs. A targeted loss multiplier (or an RL-style termination reward) is more likely to recover the ~70%+ conditional accuracy we already see at lower ratios into the "overall" column.

## 5. LLMLingua-2 and LM entropy — empirical study

The interactive `LM Entropy` tab in the demo runs a causal LM (default `Qwen2.5-0.5B-Instruct`) to compute per-token **predictive entropy** `H(p(x_i | x_<i))` and **surprisal** `−log p(x_i | x_<i)`, then pairs each word with LLMLingua-2's *actual* keep/drop decision (from `return_word_label=True`).

### Question

Does LLMLingua-2's compression signal correlate with LM entropy / surprisal?

### Observed correlations across 3 samples × 3 rates

| sample                     | Corr(keep, H)  | Corr(keep, surprisal) | kept-vs-dropped story          |
| -------------------------- | -------------- | --------------------- | ------------------------------ |
| medium · linear equation   | −0.22 to −0.24 | −0.02 to −0.10        | kept = lower entropy/surprisal |
| long · system of equations | −0.02 to −0.12 | **+0.10 to +0.18**    | kept = more surprising         |
| long · probability         | +0.02 to −0.19 | −0.08 to −0.23        | dropped = more surprising      |

**Every |r| is below 0.25, and the sign flips across samples.**

### Interpretation

1. **LLMLingua-2 is not an entropy filter.** If the method were a disguised LM-surprisal heuristic, |r| would be close to 1. We see |r| near 0 with sign flips.
2. **Sign flips are explained by repetition and phrasing.** LM surprisal rewards rare tokens. In texts with repeated content (numbers, answers, key nouns appearing more than once), the LM finds later occurrences unsurprising, but LLMLingua keeps the first instance and drops redundant repeats — which flips the direction of the correlation relative to a text with varied phrasing.
3. **Consistent with the LLMLingua-1 → LLMLingua-2 design transition.** LLMLingua-1 compressed via LM perplexity/surprisal directly. LLMLingua-2 explicitly moved to a trained classifier on GPT-4 distillation labels *because* surprisal alone doesn't capture semantic essentiality well. These empirical correlations make that design choice visible.

## 6. Open questions / next experiments

1. **Does more training fix termination at low ratios?** The §4 analysis predicts that extending to `max_steps≈4000` (≈ 3 full epochs) should raise the termination rate at ratio=0.3 and 0.1, and drag overall accuracy up toward the ~65–75% "conditional" accuracy we're already seeing when the model *does* close `</think>`. Highest-leverage next run.
2. **Would an explicit termination signal help?** Oversampling the `</think>` position in the loss (or a light length-penalty / termination-reward RL pass after SFT) would target the exact failure mode we identified, and might recover low-ratio accuracy with much less extra training than option 1.
3. **Does cosine LR still win at long horizons?** The current `constant_with_warmup` was picked to eliminate late-stage decay, but cosine often recovers last-mile accuracy on SFT runs — worth re-testing at `max_steps≈4000`.
4. **Does pointing the LM-entropy demo at the actual training model (Qwen3-4B-Thinking-2507) change the story?** The correlations reported in §5 used the small 0.5B variant. Expect numbers to shift but not the qualitative conclusion.
5. **What if we train on *all* (sample, ratio) pairs** (fan out to 4× more data) instead of randomly sampling one ratio per sample? Paper avoids this to prevent near-duplicate prompts biasing the ratio signal, but worth an ablation at the N=8 multi-sample scale — and would incidentally multiply the exposure of each `</think>` terminator.
